# One-shot protocol

## Definition

One-shot is an operator experience:

```text
one coherent product request
→ one controlled delivery process
→ one demonstrated result
```

The internal system may perform multiple bounded cycles.

## Mandatory phases

```text
CREATED
→ INTAKE
→ DISCOVERY
→ DECISIONS
→ REUSE_RESEARCH
→ SYSTEM_MAPPING
→ PLAN_COMPILE
→ PLAN_REVIEW
→ CONTRACT_LOCKED
→ IMPLEMENTING
→ VERIFYING
→ REVIEWING
→ DEMONSTRATING
→ DELIVERING
→ COMPLETE
```

Repair loops are allowed only as:

```text
REVIEWING → REPAIRING → VERIFYING → REVIEWING
```

Plan repair loops are allowed only as:

```text
PLAN_REVIEW → PLAN_COMPILE → PLAN_REVIEW
```

Both loops have fixed limits. A model cannot skip a state or declare completion.

## Decisions

MANAGEROO automatically accepts a recommended choice when it is:

- reversible;
- conventional;
- cosmetic; or
- purely internal implementation detail.

MANAGEROO stops for the operator when a decision can materially change:

- destructive data behavior;
- legal or regulatory exposure;
- recurring external cost;
- authentication/authorization boundaries;
- privacy handling;
- the product's target user or primary workflow.

This is not iterative software babysitting. It is a single consolidated product decision gate for ambiguity that cannot be safely inferred.

## Planning before coding

The complete task graph is generated and adversarially reviewed before implementation. Each task includes:

- goal;
- dependencies;
- allowed paths;
- context paths;
- acceptance;
- existing gate IDs;
- risk.

Agents may not invent commands or widen scope after locking.
